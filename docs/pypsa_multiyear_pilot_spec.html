Bare-bones multi-year PyPSA pathway model

Implementation spec for a one-zone power-only pilot covering 2030, 2040 and 2050. Linear capacity expansion and dispatch with wind, solar, gas turbines and load shedding. Solver: HiGHS. Development style: red/green TDD.

1. Scope

Build the smallest useful PyPSA model for testing multi-investment-period logic. The model is intentionally not policy-grade: no existing fleet, no CO₂ accounting, no storage, no transmission, no unit commitment.

2. Model formulation

Investment periods 2030, 2040, 2050. Period weights are configurable.
Temporal resolution 8,760 hourly snapshots per investment period.
Technologies Wind, solar, simple-cycle gas turbine, combined-cycle gas turbine, load shedding.

Decision variables

VariableDescription
CapacityEndogenous capacity for wind, solar, simple-cycle gas turbine and combined-cycle gas turbine by build year.
DispatchHourly generation from wind, solar, gas turbines and load shedding.

Objective

Minimise discounted system cost: investment cost + fixed O&M + gas variable generation cost + load shedding penalty.

Constraints

ConstraintPurpose
Hourly power balanceDemand must be met in every hour and period.
RES availabilityWind and solar dispatch limited by hourly capacity factors.
Capacity potentialsMaximum buildable capacity by technology.
Lifetime logicCapacity built in earlier periods remains active if lifetime allows.
Load sheddingHigh-cost feasibility backstop.
Important: without CO₂ cost/caps, gas may dominate if assumed gas marginal cost is low. That is acceptable for debugging but not for a transition-to-zero-emissions result.

5. Config-first assumptions

Put scenario assumptions and file references in one YAML file. Technology costs should come from the prepared TechCat Parquet files where available; gas fuel prices should be configured explicitly and combined with gas-turbine efficiency and variable O&M to form marginal costs. 

# config/model_config.yaml
investment_periods: [2030, 2040, 2050]

active_model: NL
climate_years: [1995, 2008, 2009]

model_cases:
  DK:
    demand_zones: [DKE1, DKW1]
    capacity_factor_zone: DKW1
  NL:
    demand_zones: [NL00]
    capacity_factor_zone: NL00

# Adjustable: what each modelled year represents in the objective.
period_weights:
  2030: 10
  2040: 10
  2050: 10

discount_rate: 0.04

currency: "EUR"

eurdkk: 7.47

gas_price_dkk_per_gj_fuel:
  2030: 80
  2040: 85
  2050: 85

load_shedding:
  variable_cost_eur_per_mwh: 10000
  max_capacity_mw: 1000000

capacity_limits_mw:
  wind: 50000
  solar: 50000
  gas_turbine: 100000
  gas_turbine_cc: 100000
All model costs and outputs are in EUR. Gas prices are configured in DKK/GJ fuel and converted to EUR/MWh fuel using EUR/MWh = DKK/GJ × 3.6 / EURDKK, with EURDKK = 7.47, when loading the config. The implemented model calculates gas marginal cost as converted gas price divided by electrical efficiency plus variable O&M from gas_turbine_simple_cycle_large.parquet and gas_turbine_cc_steam_extract.parquet. Annualised capital cost is derived from TechCat investment cost, fixed O&M, lifetime and discount_rate.

These files use the same schema as the gas-turbine technology file: year, electrical_efficiency_net_annual_avg, technical_lifetime_years, total_nominal_investment_meur_per_mw_e, variable_om_eur_per_mwh_e, and fixed_om_eur_per_mw_e_per_year. Demand should be prepared from National Trends Excel workbooks under DEV_DATA_DIR\TYNDP24\Demand\Demand Profiles\NT\Electricity demand profiles for 2030 and 2040, using sheets DKE1, DKW1 and NL00. The model-facing demand for the one-zone pilot is the hourly sum of the configured model_cases[active_model].demand_zones. The DK case uses DKE1 + DKW1; the NL case uses NL00. For model period 2050, reuse the 2040 demand profile.

InputRule
Wind capacity factorUse technology = onshore_wind and configured capacity_factor_zone from the PECD Parquet file. Current DK uses DKW1; current NL uses NL00.

The implemented technology loader falls back to simple built-in assumptions if the local TechCat Parquet files are missing. This keeps tests portable, but real analysis should fail loudly or report clearly when fallback assumptions are being used.

8. Implementation sketch

Build network

def build_network(cfg, data):
    import pypsa
    from pathway_pilot.technology_data import load_technology_assumptions

    tech = load_technology_assumptions(cfg)
    network = pypsa.Network()
    network.set_snapshots(data.snapshots)
    network.set_investment_periods(cfg.investment_periods)
    network.investment_period_weightings.loc[:, "objective"] = cfg.period_weights

    network.add("Bus", "electricity")
    for carrier in ["electricity", "wind", "solar", "gas", "gas_turbine_cc", "load_shedding"]:
        network.add("Carrier", carrier)

    network.add(
        "Load",
        "demand",
        bus="electricity",
        p_set=data.demand_series,
    )

    for technology, profile in [("wind", data.wind_cf_series), ("solar", data.solar_cf_series)]:
        for build_year in cfg.investment_periods:
            network.add(
                "Generator",
                f"{technology}_{build_year}",
                bus="electricity",
                carrier=technology,
                p_nom_extendable=True,
                p_nom_max=cfg.capacity_limits_mw[technology],
                capital_cost=tech[technology].capital_cost_by_period[build_year],
                marginal_cost=tech[technology].marginal_cost_by_period[build_year],
                p_max_pu=profile,
                build_year=build_year,
                lifetime=tech[technology].lifetime_years,
            )

    for build_year in cfg.investment_periods:
        network.add(
            "Generator",
            f"gas_turbine_{build_year}",
            bus="electricity",
            carrier="gas",
            p_nom_extendable=True,
            p_nom_max=cfg.capacity_limits_mw["gas_turbine"],
            capital_cost=tech["gas_turbine"].capital_cost_by_period[build_year],
            marginal_cost=tech["gas_turbine"].marginal_cost_by_period[build_year],
            build_year=build_year,
            lifetime=tech["gas_turbine"].lifetime_years,
        )

        network.add(
            "Generator",
            f"gas_turbine_cc_{build_year}",
            bus="electricity",
            carrier="gas_turbine_cc",
            p_nom_extendable=True,
            p_nom_max=cfg.capacity_limits_mw["gas_turbine_cc"],
            capital_cost=tech["gas_turbine_cc"].capital_cost_by_period[build_year],
            marginal_cost=tech["gas_turbine_cc"].marginal_cost_by_period[build_year],
            build_year=build_year,
            lifetime=tech["gas_turbine_cc"].lifetime_years,
        )

    network.add(
        "Generator",
        "load_shedding",
        bus="electricity",
        carrier="load_shedding",
        p_nom=cfg.load_shedding_max_capacity_mw,
        marginal_cost=cfg.load_shedding_variable_cost_eur_per_mwh,
    )

    # With pandas 3, convert PyPSA component string columns to object dtype
    # before optimize() so xarray can index them reliably.
    normalise_string_columns(network)
    return network

9. Preferred simple asset pattern

For clarity, use one generator per technology-build-year: 

wind_2030, wind_2040, wind_2050
solar_2030, solar_2040, solar_2050
gas_turbine_2030, gas_turbine_2040, gas_turbine_2050
gas_turbine_cc_2030, gas_turbine_cc_2040, gas_turbine_cc_2050

This makes costs, build years and lifetimes explicit. It also avoids ambiguity about whether a single extendable asset can invest repeatedly at changing costs.

12. Keep out of v0

Do not add these until the pilot solves and produces interpretable results:

FeatureReason to defer
StorageAdds time-coupled state variables and complicates interpretation.
CO₂ constraintsImportant later, but not needed for debugging pathway mechanics.
Multiple zonesTransmission and spatial data will distract from core logic.
Unit commitmentTurns a simple LP into a much harder MILP.
Endogenous learningCan require nonlinear or approximate formulations.

13. Main caveats

  • Perfect foresight: the model knows future costs and demand when investing in 2030.
  • Sparse periods: 2030, 2040 and 2050 may create lumpy build-outs.
  • No existing fleet: useful for testing, unrealistic for transition analysis.
  • No CO₂ accounting: results should not be described as decarbonisation pathways.
  • Synthetic data: fine for TDD, not for conclusions.

Recommended next step after v0: add existing fleet and CO₂ caps before interpreting results as a transition pathway.

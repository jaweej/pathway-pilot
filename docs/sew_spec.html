§1 Scope & framing

What this spec covers and the modelling assumptions behind it.

The objective is to quantify the socioeconomic welfare (SEW) effect of forcing additional capacity into the current Pathway Pilot PyPSA model. The live model is a single electricity bus with multi-investment periods, not a zonal market model. It contains extendable wind, solar, gas, and gas_turbine_cc generators for each investment period, plus fixed high-cost load_shedding. There is no offshore wind generator, transmission topology, storage, CO2 cap, or explicit emissions carrier in the current codebase.

The forcing is treated as an exogenous policy scenario. All inputs and all non-forced technologies remain identical and free to re-optimise unless a robustness stage explicitly fixes capacities.

The methodology rests on three standing assumptions, each consistent with the default PyPSA formulation:

  • Inelastic demand. Load is given exogenously from TYNDP-derived or synthetic time series; consumer surplus changes only via the price–quantity product, not via demand response.
  • Perfect competition / no market power. The LP optimum coincides with the competitive equilibrium; the shadow price of the nodal balance constraint is the wholesale price.
  • Multi-period annualised costs. capital_cost is annuitised from TechCat investment cost, fixed O&M, lifetime, and discount_rate. The PyPSA objective is weighted by period_weights, currently 10 years for 2030, 2040, and 2050.

§2 Theoretical foundation

Why welfare equals minus total system cost, up to a constant.

With inelastic demand, gross consumer benefit B is a constant independent of supply decisions. Total welfare is then

W = B − TC(x) where TC(x) is total weighted system cost and x is the vector of capacity and dispatch decisions.

Maximising W is dual to minimising TC. Comparing two runs that share the same B:

ΔSEW = Wforced − Wfree = −(TCforced − TCfree) = −(n_forced.objective − n_free.objective)

When the forced run imposes an additional binding lower bound relative to the free optimum, its feasible region is smaller and the minimum cost can only rise. The sign convention above therefore gives ΔSEW ≤ 0 for a costly forced build. If the forced capacity is below or equal to the free optimum, the constraint is non-binding and ΔSEW should be zero within solver tolerance.

§3 Two-run setup

Network preparation for the unrestricted and forced cases.

Both runs must share identical input data: active_model, demand zones, capacity-factor zone, climate/weather year, gas prices, technology costs, capacity limits, snapshot index, and investment-period weights. Only the selected generator's capacity treatment differs.

ElementRun 1 — freeRun 2 — forced
Target generator, e.g. wind_2040 True True with p_nom_min, or False with fixed p_nom
Target generator lower bound 0 MW free optimum + forced addition, or an absolute MW target
All other generators identical — extendable as in the base setup
Demand, fuel, costs, snapshots, period weights identical

For Pathway Pilot, prefer keeping the target generator extendable and setting p_nom_min. That keeps the forced asset in the same optimization family as the free run and lets the rest of the system adapt. Pinning with p_nom_min = p_nom_max is useful when testing an exact capacity target. Setting p_nom_extendable = False is also possible, but then the forced capacity is no longer part of extendable-capacity accounting.

§4 Headline metric ΔSEW

The one number that summarises the policy cost.

ΔSEW = −(n_forced.objective − n_free.objective)  ≤  0 In weighted EUR over all modelled investment periods. Divide by total forced MW, or by forced MW times represented years, depending on the reporting convention.

The current solver call uses multi_investment_periods=True and include_objective_constant=False. The objective is therefore the sum of weighted capacity and dispatch costs for the modelled periods, not a single literal calendar year. Convert to MEUR over the horizon for direct objective reporting, or divide by sum(period_weights) when a representative annual average is needed.

Sanity check
ΔSEW must be ≤ 0 to within solver tolerance when the forced lower bound is binding. A positive value indicates different inputs between runs, a modelling mistake in how the forced run was constructed, or numerical/degeneracy effects. If the free optimum already builds at least the forced level, ΔSEW should be zero.

§5 Decomposition

Breaking ΔSEW into components available from the current output schema.

The headline number is correct but uninformative. Report it alongside a cost and physical decomposition by generator and carrier. The current output layer writes optimal_capacities.parquet, hourly_dispatch.parquet, and hourly_prices.parquet; any SEW report should either add a scenario-difference writer or compute the comparison directly from solved networks before writing results.

ComponentDefinitionSign typically
ΔCAPEX Σ over generators of p_nom_opt × capital_cost, differenced run-to-run + (forcing adds capex, partly offset by crowding-out)
ΔOPEX Σ over snapshots of p × marginal_cost × weighting, differenced usually negative when wind/solar displace gas
Δload shedding penalty Dispatch from load_shedding times its configured variable cost should be zero or tiny in normal runs
Residual Difference between objective delta and the explicit decomposition small

The safest path is to compute the decomposition from solved networks using the same weights PyPSA used in the objective. For output-only reporting, read back the three parquet tables and use capital_cost, p_nom_opt_mw, marginal_cost, dispatch_mw, and price_eur_per_mwh.

Physical-quantity companion table

Alongside the cost decomposition, report:

  • Δ installed capacity by carrier and build year (MW).
  • Δ generation by carrier and investment period (TWh).
  • Δ load shedding (MWh), expected to be zero after normal solves.
  • Δ average wholesale price at the single electricity bus (EUR/MWh), by period and across the weighted horizon.
  • Δ unit overnight CAPEX exposure using unit_capex_eur_per_mw for reporting, not for the PyPSA objective.

§6 Reference implementation

Minimum viable code shaped around build_network and solve_model.

from pathlib import Path

import pandas as pd

from pathway_pilot.build_network import build_network
from pathway_pilot.config import DEV_DATA_DIR
from pathway_pilot.model_config import load_config, with_active_model
from pathway_pilot.model_inputs import build_model_inputs


def solve_network(network):
    status, condition = network.optimize(
        multi_investment_periods=True,
        solver_name="highs",
        include_objective_constant=False,
    )
    if (status, condition) != ("ok", "optimal"):
        raise RuntimeError(f"solve failed: {status=} {condition=}")
    return network


cfg = with_active_model(load_config(Path("config/model_config.yaml")), "NL")
tvar_dir = DEV_DATA_DIR / "pathway-pilot" / "tvar"
capacity_factors = pd.read_parquet(tvar_dir / "pecd_capacity_factors_2030_2040.parquet")
demand = pd.read_parquet(tvar_dir / "tyndp_demand_profiles_2030_2040.parquet")
data = build_model_inputs(
    capacity_factors=capacity_factors,
    demand=demand,
    periods=cfg.investment_periods,
    weather_year=cfg.climate_years[0],
    demand_zones=cfg.demand_zones,
    capacity_factor_zone=cfg.capacity_factor_zone,
)

n_free = solve_network(build_network(cfg, data))

target = "wind_2040"
forced_addition_mw = 1000.0
forced_level_mw = float(n_free.generators.loc[target, "p_nom_opt"]) + forced_addition_mw

n_forced = build_network(cfg, data)
n_forced.generators.loc[target, "p_nom_min"] = forced_level_mw
n_forced = solve_network(n_forced)

delta_objective = n_forced.objective - n_free.objective
delta_sew = -delta_objective

def capex(n):
    g = n.generators
    return (g.p_nom_opt * g.capital_cost).sum()

def opex(n):
    w = n.snapshot_weightings.objective
    dispatch_cost = n.generators_t.p.multiply(n.generators.marginal_cost, axis=1)
    return dispatch_cost.multiply(w, axis=0).sum().sum()

d_capex = capex(n_forced) - capex(n_free)
d_opex = opex(n_forced) - opex(n_free)
residual = delta_objective - (d_capex + d_opex)

print(f"Delta SEW: {delta_sew / 1e6:,.2f} MEUR weighted horizon")
print(f"Delta CAPEX: {d_capex / 1e6:,.2f} MEUR")
print(f"Delta OPEX: {d_opex / 1e6:,.2f} MEUR")
print(f"Residual: {residual / 1e6:,.4f} MEUR")
Note
This is deliberately kept close to the repository entry points. A production script should also write metadata matching scripts/run_pypsa_model.py: active model, demand zones, capacity-factor zone, weather year, and run timestamp.

§7 Emissions treatment

What is and is not represented in the current objective.

The current codebase does not add CO2 emissions factors, CO2 prices, or a global emissions cap to the PyPSA network. Gas marginal costs include configured fuel prices plus variable O&M only. Therefore the objective-based ΔSEW is a private system-cost metric, not a full climate-damage-inclusive welfare metric.

If emissions accounting is added later, keep the correction explicit:

ΔSEWadj = ΔSEWobj + (SCC − pCO2) × (−ΔEmissions) If forcing reduces emissions, the second term is positive. SCC is the adopted social cost of carbon.

Until that implementation exists, do not report ΔCO2 or external CO2 welfare terms as model outputs. At most, report them as an out-of-model sensitivity with clearly documented gas emissions factors and units.

§8 Model scope limits

How DK/NL cases map to the actual network.

active_model currently chooses which demand zones and capacity-factor zone are aggregated into the one-bus model. DK uses DKE1 and DKW1 demand with DKW1 capacity factors. NL uses NL00 demand and NL00 capacity factors. Both cases solve as one bus named electricity.

  • There are no cross-border flows, congestion rents, or zonal price differences to decompose.
  • The raw objective delta is the welfare-cost proxy for the selected aggregated model case, not a national market-welfare decomposition.
  • The hourly price output has one bus, so reporting should say "average electricity-bus price" rather than "DK1 / DK2 price" unless the network is extended.
Future extension
If buses, links, imports, and exports are added later, this section should be replaced with a zonal consumer-surplus, producer-surplus, and congestion-rent decomposition. That logic is not supported by the present network builder.

§9 Multiple optima & robustness

Degeneracies and the dispatch-vs-capacity decomposition.

If the unrestricted run is degenerate, the forced run's dispatch can shift in ways that are not economically meaningful but inflate apparent component changes. In this model, degeneracy is most likely among same-carrier build years with similar costs or across zero/low marginal-cost VRE assets.

  1. Tighten solver tolerances. For HiGHS, test primal_feasibility_tolerance = 1e-7 and dual_feasibility_tolerance = 1e-7, then confirm ΔSEW is stable.
  2. Run tiny smoke cases first. Use synthetic inputs with few hours per period for tests, matching the repository convention for PyPSA solve tests.
  3. Separate short-run and long-run effects. A short-run stage fixes all non-target capacities at the free optimum; the long-run stage lets all other extendable generators re-optimise. The latter is the main Pathway Pilot result.
